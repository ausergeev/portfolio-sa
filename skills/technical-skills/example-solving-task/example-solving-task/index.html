Пример решения задачи

Описание

Представим интерфейс, в котором осуществляется работа с резервуарами на предприятии. Можно добавлять резервуары. Можно указывать продукт, который будет храниться в резервуаре. Резервуар может быть не привязан к продукту (материалу). Резервуары могут быть разных размеров и 2-х видов форм (цилиндрический, прямоугольный). Можно изменять параметры резервуара и продукта. Резервуары можно удалять.

Требуется:

  1. Спроектировать REST API, необходимое для взаимодействия фронтенда и бэкенда;
  2. Описать какие данные необходимо хранить в БД;
  3. Написать SQL запрос, который вернет суммарный доступный объем в резервуарах, выделенный под каждый продукт.

Решение

Порядок действий

Решил выполнять задания в следующей последовательности:

  1. Сформировал логическую модель данных, что должно помочь понять, какими данными должны обмениваться фронт и бэк при проектировании REST API;
  2. Написал SQL запрос на основе разработанной модели данных;
  3. Спроектировал REST API с учётом разработанной модели данных.

Модель данных

Нарисовал в Draw.io логическую модель данных в нотации Мартина. В связи с отсутствием НФТ, которые могли бы требовать денормализации, выполнил нормализацию модели данных до 3-й нормальной формы. Для этого:

  1. Выделил сущности;
  2. Выделил атрибуты сущностей;
  3. Ввёл суррогатные ключи (т.к. не смог выделить ключи из атрибутов);
  4. Определил модальность и кардинальность связи между таблицами (у одного резервуара может быть 0 или 1 продукт; продукт может быть в 0 или множестве резервуаров).

Для разработки и апробации SQL запроса принял решение развернуть БД на основе разработанной модели данных, наполнить её тестовыми данными и продемонстрировать работу запроса. Предположил, что список возможных форм резервуаров впоследствии может быть расширен, в связи с чем решил задать ограничение по длине как для атрибута названия. Для подготовки SQL файла, позволяющего создать необходимые таблицы и зависимости в PostgreSQL, воспользовался сервисом https://dbdiagram.io Для визуализации физической модели данных c использованием DBML подготовил следующую структуру:

// Use DBML to define your database structure

Table reservoirs {
  reservoir_id uuid [primary key]
  name VARCHAR(255)
  size integer
  form VARCHAR(255)
  product_id uuid
}
Table products {
  product_id uuid [primary key]
  name VARCHAR(255)
}
Ref: reservoirs.product_id > products.product_id // many-to-one

С помощью сервиса https://dbdiagram.io подготовил SQL файл для PostgreSQL, позволяющий создать в БД необходимые таблицы и связи между ними:

CREATE TABLE "reservoirs" (
  "reservoir_id" uuid PRIMARY KEY,
  "name" VARCHAR(255),
  "size" integer,
  "form" VARCHAR(255),
  "product_id" uuid
);
CREATE TABLE "products" (
  "product_id" uuid PRIMARY KEY,
  "name" VARCHAR(255)
);
ALTER TABLE "reservoirs" ADD FOREIGN KEY ("product_id") REFERENCES "products" ("product_id");

Таким образом, как минимум один резервуар не привязан к продукту. Как минимум один продукт отсутствует в каком-либо резервуаре. Этих данных посчитал достаточными для отработки запроса. Теперь для подготовки самого SQL запроса на реальной БД выполнил декомпозицию задачи:

  1. Выполнил джойн, т.к. нужные нам данные находятся в разных таблицах;
  2. Сгруппировал записи по продукту и применил агрегирующую функцию суммирования по объему;
  3. Принял NULL за 0, отсортировал результат по объему в порядке убывания.

Таким образом, SQL запрос выглядит так:

SELECT p.name AS name_product, SUM(COALESCE(r.SIZE,0)) AS sum_size
FROM reservoirs AS r
RIGHT OUTER JOIN products AS p ON p.product_id = r.product_id
GROUP BY p.name
ORDER BY sum_size DESC;

            example: "Невалидное значение параметра"
          parameter:
            type: string
            description: "Название параметра, где обнаружена ошибка"
            example: "form"

Примечания к заданию:

  • Для изменения параметров резервуара и продукта использовал метод PATCH, а не PUT, поскольку именно PATCH подходит для обновления существующих ресурсов и позволяет сократить объем трафика;
  • Для product_id задал возможность иметь значение null для случая, когда нет нужды связывать резервуар и продукт при создании резервуара или когда необходимо отвязать продукт при изменении параметров резервуара;
  • Предположил, что объемы справочников невелики и нет нужды делать пагинацию, сортировку и фильтрацию по параметрам;
  • Предположил, что ПО делается для работы в российском сегменте, потому описание ошибок сделал на русском языке, чтобы в последующем была возможность их тексты пробросить как есть в GUI.
  • Предположил, что в GUI возможно единовременное создание только одного экземпляра резервуара, поэтому в теле запроса POST /reservoirs указал параметры одного резервуара;
  • Упростил функционал удаления резервуара до одного запроса DELETE /reservoirs/{reservoirid} без возможности удаления множества резервуаров (для чего понадобился бы уже отдельный POST запрос).
